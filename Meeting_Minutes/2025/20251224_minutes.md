2、rust和 golang vendor cache 分析进展汇报，及版本差异合并讨论 -- 胡欣蔚

  2.1 已分析 openEuler 上 src-openeuler 的代码仓，可通过 API 正常访问的代码仓中，确定没有 golang/rust 的 9258 个，疑似有 golang/rust 依赖的 176 个。折算相关代码长占比整体不超过 2%。

  2.2 golang 抽样了git-lfs, grafana, cpds-analyzer三个包。共引入 535 个 golang 组件（同版本重复计算），其中 31 个组件版本相同，45 个组件版本不同。其中 grafana 是大型软件，除了 golang 之外，还有 yarn 的组件依赖引入。

  2.3 rust 选了 utshell, which-rs, dust 三个包。共引入 143 个 Cargo 组件（同版本重复计算），其中 5 个组件版本相同，25 个组件版本不同。 版本差异看起来更多是构建引入时间差异。这三个组件体量相当，不同版本所占比例感觉有一定代表性。

  2.4 后续计划
   - 选项一：保持现状，在 spec 中注入对golang和rust组件的构建依赖，表明软件成分。增加可追溯能力，对各个组件当前不做额外要求。
   - 选项二：尝试部分通过 -devel 方式打包建仓，主要解决不同架构的非upstream改进问题。会出现golang和rust组件两种形态共存的情况，可能会造成一定的混乱。工作量不大
   - 选项三：对golang 和 rust 组件统一做打包管理，每个组件以 -devel.rpm 的形式提供，作为 BuildRequires 加入 软件包 spec 中。估计需要给 openEuler 增加 上千个代码仓。影响比较大。
   - 待分析：golang 和 rust 组件的安全漏洞披露及修复，对 openEuler 的影响有多大。
